# 玩家客户端收口计划

## 目标

在不破坏现有开发效率的前提下，把当前 `bevy_debug_viewer` 从“调试器”收口为“可给玩家使用的临时正式客户端”，或者为未来独立玩家端做清晰过渡。

## 当前现状

- `bevy_debug_viewer` 已经承担了实际 playable 入口的大部分职责。
- 它同时包含：
  - 地图展示
  - 交互控制
  - 背包、技能、制作、交易 UI
  - 信息面板
  - 调试信息
  - 控制台
- 这对开发有价值，但会干扰玩家理解。

## 关键问题

- 玩家端和调试端职责混杂。
- 调试用状态、面板和命令会污染默认体验。
- 当前缺少明确规则说明哪些 UI 属于玩家，哪些只属于开发者。

## 本阶段范围

- 不强制立即新建 `bevy_game` app。
- 先定义 `bevy_debug_viewer` 如何被收口成“可玩构建入口”。
- 若后续需要独立 app，再以该规则平移。

## 不做什么

- 不在本阶段重写 UI 框架。
- 不为调试与玩家分别维护两套 runtime。
- 不引入新的并行入口却长期双维护。

## 收口原则

### 默认玩家可见

- 主菜单
- 新游戏/继续
- 必要 HUD
- 背包、角色、技能、制作、地图
- 对话与交互提示
- 任务与状态提示
- 失败恢复提示

### 默认开发隐藏

- 高频调试信息面板
- AI 内部状态面板
- 性能调试面板
- 控制台命令入口
- 仅用于观察 runtime 内部的视图

### 可选开发开关

- 如果仍需要保留在同一 app 内，则用明确开关暴露，而不是默认展示。

## 需要明确的产品流程

- 主菜单进入游戏
- 暂停/设置
- 新游戏与继续游戏
- 死亡/失败后的处理
- 返回主菜单或重新开始

这些流程若仍依赖控制台命令或开发者认知，则还不算完成。

## 结构建议

近期建议：

- 继续以 `bevy_debug_viewer` 作为运行入口。
- 用配置或构建模式区分 `developer` 和 `player` 可见功能。
- 将 debug 面板和控制台明确归类为开发辅助层。

中期建议：

- 若玩家入口越来越稳定，再考虑抽出独立 `bevy_game` app。
- 该 app 仍然消费同一套 `game_core`、`game_bevy`、`game_data`，不重写逻辑。

## 依赖模块

- `rust/apps/bevy_debug_viewer/src/app.rs`
- `rust/apps/bevy_debug_viewer/src/game_ui`
- `rust/apps/bevy_debug_viewer/src/info_panels`
- `rust/apps/bevy_debug_viewer/src/console.rs`
- `rust/crates/game_bevy`

## 任务拆分

### A. 定义玩家默认界面

- 哪些面板必须保留。
- 哪些信息属于玩家可理解信息。
- 哪些面板默认隐藏。

### B. 收口主流程

- 新游戏
- 继续游戏
- 暂停/设置
- 失败恢复

### C. 隔离开发辅助能力

- 控制台
- AI/性能/内部状态面板
- 仅用于验证 runtime 的临时按钮和状态

## 验收标准

- 不看源码也能理解默认界面。
- 新玩家默认看不到与内部调试强绑定的噪音。
- 玩家关键流程都能通过正式 UI 完成。
- 开发者仍能通过开关进入调试模式，不丢失开发效率。

## 风险

- 如果继续长期把 debug UI 当正式 UI 使用，后续每加一个调试能力都会污染玩家端。
- 如果过早单独拉一套玩家端 app，又可能导致重复维护。
