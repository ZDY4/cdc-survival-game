# 任务、对话与成长挂接计划

## 目标

把现有任务系统从“可用能力”变成“真实驱动层”，让任务、对话、奖励、地点解锁和成长形成稳定链路。

## 当前现状

- `game_core` 已有任务开始、推进、完成与奖励能力。
- `game_core` 对话执行中已经支持 `start_quest` 等 action。
- UI 层已经能展示任务标题列表。
- 数据层已经有 6 个任务和 7 份对话内容。
- 现有任务推进偏少，已确认至少有 `kill` 和 `collect` 两条推进路径。

## 关键问题

- 任务可以存在，但不一定会被新游戏或对话稳定发放。
- 任务 objective 类型与 runtime 推进逻辑不完全一致。
- 奖励虽然支持经验、物品、技能点、配方解锁，但缺少针对玩家体验的统一设计。
- 对话既承担叙事，又承担任务流转，当前缺少明确规范。

## 本阶段范围

- 定义首批“可正式使用”的 objective 类型。
- 统一任务发放入口。
- 统一对话动作约定。
- 统一任务奖励节奏。
- 建立最小任务内容约束。

## 不做什么

- 不在本阶段设计复杂分支任务编辑器。
- 不引入一套脱离 `game_data` 的任务脚本系统。
- 不把任务规则塞回前端 UI。

## 建议的 objective 白名单

建议先只把以下目标作为第一批正式支持对象：

- `collect`
- `kill`
- `talk`
- `return`
- `enter_location`

在这些都跑通之前，先不要继续扩更多 objective 类型。

对于 `sleep`、`repair`、`craft_at_station` 这类目标：

- 要么补 runtime 推进逻辑。
- 要么暂时不要用于主线任务。

## 现实约束

基于当前代码阅读，本轮先按以下事实推进：

- 已确认 runtime 存在 `kill` 和 `collect` 推进路径。
- 对话 action 已支持 `start_quest`。
- 尚未把 `sleep` 视为可依赖的主线 objective。

因此近期主线任务不要再使用未确认落地的 objective 类型。

## 对话挂接规则

### 发任务

- 首次发任务只通过对话 action 或新游戏开局脚本之一完成。
- 同一任务不要同时依赖多个入口自动发放。

### 交任务

- 交付任务时优先通过对话结果驱动奖励和下一步。
- 若对话负责交付，必须让 UI 明确告诉玩家“谁能交”。

### 解锁地点

- 新地点解锁应作为任务奖励或对话结果出现。
- 不建议在开局一次性解锁过多地点。

## 奖励设计规则

首批任务奖励控制在下列范围内：

- 经验
- 少量物资
- 1 个配方解锁
- 1 个地点解锁
- 关系或信任值变化，前提是后续系统有消费场景

不要让首个任务奖励过重，避免破坏后续搜刮、交易和制作节奏。

## 内容约束

每个任务至少写清楚：

- 发起者是谁
- 玩家第一步去哪
- 完成条件是什么
- 回来找谁交
- 完成后得到什么
- 下一步动机是什么

每份承载任务流转的对话至少写清楚：

- 何时出现
- 是否重复可见
- 是否受任务状态影响
- 是否触发 `start_quest`
- 是否触发奖励或地点解锁

## 验收标准

- 首个主线任务可稳定发放。
- 任务目标可被实际推进，不存在“UI 有任务但永远不会完成”的情况。
- 交任务后能稳定获得奖励与下一步驱动。
- 任务内容和对话内容的责任边界清晰。

## 风险

- 若继续在任务内容中使用 runtime 未支持的 objective，任务链会反复卡死。
- 若多个入口都能发同一任务，容易出现重复发放或状态错乱。
- 若对话分支不绑定任务状态，玩家可能看到不一致文本。

## 本 plan 完成定义

当以下条件同时满足时，可认为任务与对话挂接已经进入可执行状态：

- 首条任务链的 objective 类型已与 runtime 支持范围对齐。
- 首条任务链只有一个发放入口和一个稳定交付入口。
- 任务奖励、地点解锁和下一步驱动都能通过数据内容清晰表达。
