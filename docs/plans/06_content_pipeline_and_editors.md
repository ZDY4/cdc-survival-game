# 内容生产管线与编辑器计划

## 目标

让地图、角色、任务、对话、配方等内容可以稳定扩展，并且继续以共享 Rust 层为权威，不让编辑器前端重新定义长期 schema。

## 当前现状

- `game_data` 已承担大部分内容 schema、加载与校验。
- 仓库内已经有独立编辑器方向：
  - `bevy_map_editor`
  - `bevy_character_editor`
  - `bevy_item_editor`
  - `bevy_recipe_editor`
  - `tools/tauri_editor`
- 这说明基础方向是对的，但还需要把“内容如何稳定进入可玩版本”定义清楚。

## 核心问题

- 当前内容已经不少，但缺少一套面向 playable 里程碑的生产流程。
- 一项内容往往跨越地图、任务、对话、掉落、商店、配方，容易散落修改。
- 如果不定义“哪个工具负责什么”，编辑器边界会继续模糊。

## 本阶段范围

- 只定义内容生产流程和边界。
- 不在本阶段大规模扩编辑器功能。
- 优先服务首个 playable 里程碑所需内容。

## 不做什么

- 不让编辑器前端成为运行时权威。
- 不做多套 schema 并行维护。
- 不为每类内容马上造一套复杂工作流系统。

## 内容生产建议按职责分层

### 共享 Rust 层

职责：

- schema
- 加载
- 校验
- 基础编辑服务

对应目录：

- `rust/crates/game_data`
- `rust/crates/game_editor`

### Bevy 编辑器

职责：

- 地图、空间布局、场景预览、需要 3D 交互的编辑任务

对应目录：

- `rust/apps/bevy_map_editor`
- `rust/apps/bevy_character_editor`
- 其他 Bevy 侧专用编辑器

### 独立编辑器

职责：

- 任务、对话、表单型内容、工作流型内容

对应目录：

- `tools/tauri_editor`

## 可玩里程碑下的推荐内容包组织

建议以后围绕“一个内容包”来推进，而不是散着改：

- 一个地点包
- 一个任务包
- 一个 NPC 包
- 一个资源循环包

例如“超市内容包”至少应包含：

- `map`
- 对应敌人/掉落
- 任务接入
- 对话文案
- 奖励与解锁

## 建议的轻量流程

1. 在 plan 文档里确定目标内容包。
2. 明确要修改的数据域。
3. 分别用对应编辑器或数据文件完成修改。
4. 通过 `game_data` 校验和 `cargo check` 验证。
5. 用 `bevy_debug_viewer` 做最小 smoke。

## 依赖模块

- `rust/crates/game_data`
- `rust/crates/game_editor`
- `rust/apps/bevy_map_editor`
- `rust/apps/bevy_character_editor`
- `tools/tauri_editor`

## 任务拆分

### A. 建立“内容包”方法

- 用地点或任务链作为组织单位。
- 每次迭代至少明确涉及哪些数据目录。

### B. 定义工具边界

- 空间内容优先走 Bevy 编辑器。
- 表单型叙事内容优先走独立编辑器。
- 通用校验统一落到共享 Rust 层。

### C. 加强进入 playable 的验证链

- 内容提交前至少做 schema 校验。
- 每个关键内容包至少做一次运行时 smoke。

## 验收标准

- 新增一段任务或一个地点时，团队知道应该改哪些目录和用哪个工具。
- 内容最终仍由共享 Rust 层消费和校验。
- 不出现编辑器各自维护长期漂移 schema 的情况。

## 风险

- 如果继续散着改数据文件，内容扩展速度会越来越慢。
- 如果编辑器边界不清晰，后续会出现重复实现和权威漂移。
