# Agent 游戏设计参考

本文面向开发 agent，用于快速理解本项目的玩法定位和参考游戏边界。它不替代 `AGENTS.md` 的工程规则；涉及具体代码落点、数据权威来源和验证方式时，仍以 `AGENTS.md` 为准。

## 项目定位

本项目是一个传统类 Rogue 的即时回合制生存探索游戏。玩家在网格化或离散空间中探索、战斗、搜刮、交互和推进任务；世界在玩家或其他行动者执行动作时推进，而不是依赖纯实时反应。

开发时可以把核心体验理解为：

- 玩家做出一个明确动作。
- core 层根据规则计算时间、行动消耗、风险、战斗、交互和资源变化。
- 存档、任务、背包、角色状态或地图对象产生可追踪的结果。
- world 和 ui 层只表现结果，不自行决定玩法结果。

## 即时回合制语义

这里的“即时回合制”不是动作游戏式实时战斗，也不是棋盘式固定阵营轮流。它更接近传统 roguelike 的行动驱动时间模型：

- 玩家输入移动、等待、攻击、拾取、使用、交谈、开关门、搜索等离散动作。
- 每个动作有时间、行动点、噪音、风险或冷却等代价。
- NPC、敌人、环境事件和状态效果按行动消耗或世界时间推进。
- 当玩家停止行动时，世界不应因为真实时间流逝而强制快速推进，除非特定系统明确设计为实时提示或表现。
- UI 可以有动画、过渡和反馈，但动画不应改变 core 的判定结果。

## 参考游戏

### Stoneshard

参考方向：

- 俯视角探索、格子移动和行动驱动的战斗节奏。
- 生存压力、伤病、补给、负重和旅途风险带来的长期决策。
- 小尺度战斗中的位置、距离、消耗品和状态管理。
- 场景探索和资源搜刮与角色成长之间的循环。

不应照搬：

- 具体世界观、职业、技能树、数值公式或 UI 结构。
- 高惩罚体验本身不是目标，除非与本项目的 CDC 生存主题和任务结构匹配。

### The Doors of Trithius

参考方向：

- 开放探索、城镇/野外/地下区域之间的冒险节奏。
- 任务、地点、派系、商人、容器和动态遭遇组合出的可持续探索。
- 传统 roguelike 操作感与较现代的地图、日志、交互提示结合。
- 内容驱动的世界扩展方式：地点、NPC、任务和物品应能通过数据与场景逐步增长。

不应照搬：

- 奇幻设定、装备体系或地牢生成规则。
- 任何与本项目 Godot scene 地图权威来源冲突的长期地图数据流。

### Caves of Qud

参考方向：

- 高系统性世界：角色状态、物品、环境、阵营、任务和突发事件可以组合出复杂结果。
- 文本信息、日志、检查面板和交互反馈对复杂规则的解释能力。
- 开放探索中允许玩家用多种手段解决问题，而不是只沿单一路径推进。
- 规则优先的 emergent gameplay：表现层展示结果，核心层统一裁定结果。

不应照搬：

- 超高复杂度的突变、身体部位、程序生成生态或长文本密度。
- 为追求系统深度而牺牲当前项目的可维护架构边界。

## 本项目应吸收的共同点

- 行动驱动：玩家动作是世界推进的主要节拍。
- 离散规则：移动、交互、战斗、搜刮和等待都应能映射到明确规则。
- 风险管理：补给、伤病、感染、噪音、时间、负重、可见性和敌对行为都可以成为策略压力来源。
- 信息反馈：复杂结果必须通过 HUD、日志、提示或检查面板解释清楚。
- 数据扩展：物品、任务、角色、容器、地点和事件应优先沿数据层与既有 scene 工作流扩展。
- 分层实现：玩法结果在 `godot/scripts/core`，内容读写在 `godot/scripts/data`，地图表现和 UI 只负责展示与输入转发。

## 本项目不追求的方向

- 不做纯实时动作战斗。
- 不以反应速度、走位手感或连招输入作为核心挑战。
- 不把参考游戏的具体系统、数值、世界观或美术风格直接复制进项目。
- 不因为参考传统 roguelike 就接受不可读、不可测、不可维护的规则堆叠。
- 不把地图主来源重新退回长期 JSON 工作流；地图开发仍以 `godot/scenes/maps/*.tscn` 为主。

## 开发落点提醒

常见设计需求应优先落到以下位置：

| 需求 | 优先落点 |
| --- | --- |
| 行动消耗、等待、移动、战斗、状态结算 | `godot/scripts/core` |
| 物品、任务、角色、容器、交易等非地图内容 | `data/` + `godot/scripts/data` |
| 地图布局、入口点、地图对象 | `godot/scenes/maps/*.tscn` |
| 地图对象和快照的可视化 | `godot/scripts/world` |
| HUD、日志、面板、提示、交互按钮 | `godot/scripts/ui` |
| 启动装配、输入转发 | `godot/scripts/app` |
| 编辑器体验和内容 handoff | `godot/addons/cdc_game_editor` |

如果一个功能同时涉及规则、数据和表现，先明确 core/data 的权威结果，再让 world/ui 展示这些结果。不要让 UI、world 或 editor dock 自行发明第二套判定。

## Agent 判断准则

当任务描述只说“做得更像 roguelike”或“参考 Stoneshard / The Doors of Trithius / Caves of Qud”时，agent 应先转译为具体工程问题：

- 这是动作推进、时间消耗还是状态结算问题？
- 这是内容数据、地图 scene、核心规则、UI 反馈还是编辑器体验问题？
- 是否需要新增数据 schema 或 validator？
- 是否会扩大 `game_app.gd` 的业务职责？
- 是否绕过了 Godot 数据层或核心层？

只有当这些问题有明确答案后，再开始修改代码或内容。
