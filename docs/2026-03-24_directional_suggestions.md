# 2026-03-24 Directional Suggestions

本文件保留原 2026-03-24 四份文档中仍然有价值、但还不到“可以直接开发排期”的内容，包括方向性建议、架构原则、风险提醒、推荐顺序与验收视角。

## 1. 总体方向

- 继续保持三端分离：
  - `逻辑` 在 Rust / Bevy
  - `表现` 在 Godot
  - `编辑` 在独立工具
- 不要为了短期方便把未来属于 Rust / Bevy 的核心逻辑重新耦合回 Godot
- 保持共享 Rust 模型作为内容 schema 与运行时交换结构的权威来源

## 2. Bevy 交互方向建议

- 优先补权威运行时，再补协议接入，最后补体验层
- Godot 若重新接入，应作为协议客户端，而不是重新持有交互权威
- 后续如果继续沿当前路线推进，推荐顺序仍然是：
  1. 权威 `DialogueRuntimeState`
  2. 真实地图切换
  3. `bevy_server` 本地 TCP transport
  4. Godot 协议前端

## 3. Item Editor 方向建议

- 继续把可共享约束下沉到 Rust 层，避免编辑器长期维护另一套规则
- 不在这一阶段再次大幅推翻 item schema
- 当前更适合在现有 fragment schema 上增强内容制作效率，而不是重新设计模型
- 剩余任务的推荐顺序：
  1. 模板与复制工作流
  2. lint / report
  3. registry / catalog 收紧
  4. fragment 头部快速编辑
  5. 更多测试

### Item Editor 成功标准参考

- 内容作者能独立完成常见 item 新建、复制、配方与效果配置
- 常见引用错误无需离开当前字段即可定位
- 中等规模 item/effect catalog 下仍能快速搜索和编辑
- 不增加对 Godot 运行时表现层的新耦合

## 4. NPC AI 架构建议

- 长期坚持 `Utility AI -> Light GOAP -> FSM / Executor` 三层结构
- `Utility` 只负责“选目标”，不直接输出动作链
- `GOAP` 只处理离散事实和短计划，不直接理解整个连续世界
- `Executor` 只负责执行，不回收规划职责

### 不建议做的事

- 不要让 GOAP 继续膨胀成全能系统
- 不要把目标选择或规则判断重新写回 Godot
- 不要过早把动作系统完全数据化、编辑器化
- 不要让 `NpcPlanRequest` 无限膨胀

### 结构优化方向

- 逐步把评分逻辑拆成独立评分项
- 逐步让职业差异通过“目标权重 + 动作集变化”表达
- 长期可考虑：
  - `NpcUtilityContext`
  - `NpcPlanningContext`
  - 按职业划分动作模块
  - 独立 reservation 服务

## 5. 推荐推进顺序

### NPC AI

推荐顺序：

1. 继续巩固 `Utility` 结构
2. 扩展下一个职业样板
3. 增强调试可视化
4. 提升在线执行真实性
5. 接入 Godot 只读同步
6. 最后做编辑器支持

### 交互迁移

推荐顺序：

1. 共享运行时权威化
2. 协议与 transport
3. 前端接入
4. 调试体验增强

## 6. 风险提醒

### 交互迁移

- 如果 Godot 再次扩张交互权威，三端分离会倒退
- 如果 viewer 与正式前端维护两套对话推进逻辑，行为会继续分叉

### NPC AI

- 如果目标选择与动作拆解重新混写，后续职业复用会变差
- 如果 GOAP 开始承载导航、动画、复杂社交、连续值细节，可解释性和性能都会恶化
- 如果动作域还没稳定就过早做编辑器配置化，schema 会反复变化

## 7. 验证视角建议

### NPC AI

- 单元测试继续关注：
  - 连续需求到离散事实转换
  - Utility 目标选择
  - GOAP 计划生成
  - reservation 冲突重规划
  - durative action 生命周期
- 集成测试继续关注：
  - 单角色完整日常
  - 多角色岗位覆盖
  - 用餐窗口切换
  - 夜间回家睡觉
  - 警报打断
  - 玩家不在场时的离线推进

### 交互与编辑器

- 交互侧优先验证“共享运行时是否成为唯一权威”
- 编辑器侧优先验证“引用、校验、保存、重载是否保持一致”
