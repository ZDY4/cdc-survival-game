# 新架构方案草案

## 目标

将当前末日生存 RPG 改造为三端分离架构，在保证游戏性能的同时，获得更灵活的内容生产与编辑能力。

- 游戏逻辑高性能运行
- 编辑器界面现代、可定制、易扩展
- 游戏、编辑器、数据链路统一

## 落地策略

整体目标仍然是：

`Bevy（逻辑核心） + Godot（渲染客户端） + Tauri 2 Web（独立内容编辑器）`

但实施上不建议一步到位重构，而是分阶段推进。

### 第一阶段：建立 Rust 基础层

- 搭建 `rust/` workspace
- 抽出共享数据模型、协议定义、校验逻辑
- 先不大规模改动现有 Godot 运行结构

这一阶段的目标是先把“共享数据源”建立起来。

### 第二阶段：引入 Bevy 逻辑服务

- 新建 `bevy_server`
- 先接管适合服务化的核心逻辑
- 通过 `IPC / TCP` 与 Godot 通信

这一阶段的目标是逐步把核心规则从 Godot 挪出，而不是一次性迁移全部系统。

### 第三阶段：编辑器独立化

- 将 `addons/cdc_game_editor` 中的编辑能力逐步迁出
- 在 `tools/tauri_editor` 中实现独立编辑器
- 让编辑器直接复用 Rust 侧共享数据与校验逻辑

这一阶段的目标是形成统一的数据生产链路。

### 长期目标：形成稳定三端架构

- `Bevy` 负责核心逻辑、ECS、AI、寻路与状态计算
- `Godot 4` 负责 2D 渲染、UI、动画与输入
- `Tauri 2 + Web` 负责内容编辑与工具界面

通信统一使用本地 `IPC / TCP`，支持多个客户端同时连接 `Bevy` 逻辑核心。

## 设计重点

### 1. 逻辑与表现彻底分离

- `Godot` 不承载核心游戏逻辑
- `Bevy` 不负责渲染
- 编辑器不直接耦合运行时表现层

这样可以降低模块耦合，提升开发、调试和后续维护效率。

### 2. 数据结构统一

所有游戏内容数据统一由 `Rust` 结构体定义，供 `Bevy` 与编辑器共享：

- 避免重复定义
- 避免格式不一致
- 降低导入导出与兼容成本

### 3. 编辑器独立化

当前 `addons/cdc_game_editor` 中的编辑能力，逐步迁移为独立编辑器。

目标能力包括：

- 道具编辑
- 对话与剧情编辑
- 多层地图编辑
- 数据导入 / 导出
- 自动校验与错误提示

界面采用 Web 技术实现，以便获得更高的 UI 自由度与更好的交互体验。

## 三层架构

```text
┌─────────────────────┐    ┌─────────────────────┐    ┌─────────────────────┐
│  独立内容编辑器      │ ←→ │  Bevy 逻辑核心      │ ←→ │  Godot 渲染客户端    │
│  Tauri2(Web+Rust)   │ IPC │ 无头/窗口模式       │ IPC │  2D 渲染 / UI / 输入 │
└─────────────────────┘    └─────────────────────┘    └─────────────────────┘
          ↑                           ↑
          └──────────── 共享 Rust 数据模型 ────────────┘
```

## 各端职责

### Bevy 逻辑核心

技术栈：

- `Bevy`
- `Tokio`
- `Serde`

主要职责：

- 管理玩家、怪物、道具、障碍物、交互物等实体
- 承载 AI、寻路、战斗、状态计算等核心逻辑
- 作为运行时数据与通信中心
- 为 `Godot` 和编辑器提供 IPC 服务

附加能力：

- 支持无头模式运行，便于发布
- 支持窗口模式运行，便于开发调试

### Godot 渲染客户端

技术栈：

- `Godot 4`

主要职责：

- 2D 画面渲染
- UI 与动画表现
- 玩家输入处理
- 作为 IPC 客户端接收逻辑层状态并进行展示

### 独立内容编辑器

技术栈：

- `Tauri 2`
- `Rust`
- `React`
- `Tailwind CSS`

主要职责：

- 编辑游戏内容
- 管理与校验数据
- 导出 / 导入 JSON 数据
- 连接运行中的 `Bevy` 进程进行实时预览

## 实施流程

### 日常开发

- 现阶段仍以 Godot 项目为主要开发入口
- Rust 模块按需独立开发与验证
- 编辑器能力按模块逐步迁出，不要求一次替换完成

### 联调模式

- 运行 `Bevy` 调试进程
- 运行 `Godot` 渲染客户端
- 运行独立编辑器
- 三端通过 `IPC` 联调，实现内容同步与实时预览

### 发布模式

- `Bevy` 以无头模式作为纯逻辑服务运行
- `Godot` 打包为正式游戏客户端
- 编辑器作为独立桌面工具单独分发

### 推荐推进顺序

1. 先统一数据模型与协议
2. 再迁移适合服务化的核心逻辑
3. 最后替换现有编辑器链路

这样可以把风险拆小，避免在渲染、逻辑、工具三条线上同时重构。

## 核心约束

- 逻辑唯一归属 `Bevy`，不要把核心规则写进 `Godot`
- 渲染唯一归属 `Godot`，不要在 `Bevy` 中承担表现职责
- 数据唯一来源是共享 `Rust` 数据模型，禁止重复定义
- 通信统一走 `IPC / TCP`，避免额外耦合通道
- 内容编辑逐步从 Godot 插件迁移到独立编辑器

## 建议目录结构

为了降低迁移成本，建议采用渐进式演进，而不是一次性把现有 Godot 工程整体搬进新目录。

推荐做法是：

- 短期保留当前 Godot 项目继续位于仓库根目录
- 新增 Rust workspace，逐步承接共享数据与逻辑服务
- 将编辑器独立到工具目录，再逐步替换现有 Godot 插件编辑能力

```text
cdc_survival_game/
├── project.godot              # 当前 Godot 项目入口，暂时保留在根目录
├── scenes/                    # Godot 场景
├── core/                      # 当前共享框架层
├── modules/                   # 当前功能模块
├── systems/                   # 当前全局系统
├── assets/                    # 美术、音频、字体等资源
├── data/                      # 物品、对话、地图、配方等内容数据
├── addons/                    # 现有 Godot 插件，后续逐步瘦身
├── tools/
│   └── tauri_editor/          # 独立内容编辑器
│       ├── src/               # Web 前端
│       ├── src-tauri/         # Tauri Rust 后端
│       └── tauri.conf.json
├── rust/
│   ├── Cargo.toml             # Rust workspace 配置
│   ├── crates/
│   │   ├── game_data/         # 共享数据模型
│   │   ├── game_protocol/     # IPC 协议
│   │   └── game_core/         # 公共逻辑与校验能力
│   └── apps/
│       └── bevy_server/       # Bevy 逻辑核心
│           ├── Cargo.toml
│           └── src/
└── tests/                     # 测试与验证脚本
```

这个结构更适合当前项目逐步演进：

- Godot 侧保持稳定，避免大量 `res://` 路径调整
- Rust 侧可以独立成长，不打断现有开发节奏
- 编辑器先独立，再逐步接管内容生产流程

## 一句话总结

这套方案的核心不是“换技术栈”，而是把 `逻辑`、`表现`、`编辑` 三件事拆开，让性能、工具体验和后续扩展都更可控。
