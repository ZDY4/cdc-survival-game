# AGENTS.md

本文件定义本仓库内 agent 的默认工作方式、架构边界与实施优先级。

## 项目目标

本项目正从当前单体式 `Godot` 末日生存 RPG，逐步演进为三端分离架构：

- `Bevy` 负责核心逻辑
- `Godot 4` 负责渲染、UI、动画与输入
- `Tauri 2 Web` 负责独立内容编辑器
- 共享 `Rust` 数据模型作为内容 schema 与跨端交换结构的统一来源
- 端间通信统一使用本地 `IPC / TCP`

目标是将功能实现与长期演进基准统一收敛到 `Rust / Bevy` 端；`Godot` 仅作为渲染与输入表现层存在，不再作为功能实现兼容基准。

## 实施顺序

默认按以下顺序推进，优先把功能与数据权威收敛到 `Rust` 端：

1. 建立 `rust/` workspace，先抽出共享数据模型、协议定义与校验逻辑。
2. 新建 `bevy_server`，逐步接管适合服务化的核心逻辑。
3. 将 `addons/cdc_game_editor` 的编辑能力迁移到 `tools/tauri_editor`。
4. 最终形成稳定的三端协作架构。

若任务与以上顺序冲突，优先选择更小、更直接、且能继续强化 `Rust` 权威实现的改动。

## 架构边界

所有 agent 在设计、编码、重构时都必须遵守以下边界：

- 核心游戏规则、状态计算、AI、寻路、战斗逻辑，长期归属 `Bevy` 侧。
- `Godot` 侧负责渲染表现、UI、动画、输入，不应继续扩大核心规则承载范围。
- 编辑器能力应从 `addons/cdc_game_editor` 迁出，目标是独立桌面工具。
- 游戏内容数据的结构定义必须统一到共享 `Rust` 模型，作为内容 schema 与跨端交换结构的唯一基准，避免多端重复定义。
- 进程或模块间通信优先采用统一协议与 `IPC / TCP`，避免临时性私有耦合方案。
- `game_bevy` 可以依赖 `Bevy` 和 `game_data` / `game_core`，用于承载多个 `Bevy` app 共享的运行时装配；但它不承载纯规则，也不替代 `game_core`。

## Bevy / Rust 约束

- 迁移到 `rust/apps/bevy_server` 的系统，以及长期在 `Bevy` 侧新增的运行时逻辑，应优先按 `Bevy ECS` 方式组织，使用 `Component`、`Resource`、`Event`、`System`、`Plugin` 建模与调度。
- 不要把 `Godot` 侧现有节点脚本、全局管理器或大型面向对象控制器直接平移到 `Bevy`。
- 放在 `rust/crates/game_data`、`rust/crates/game_protocol`、`rust/crates/game_core` 的共享层逻辑，应优先保持引擎无关或仅弱依赖特定引擎，以便被 `Bevy` 与编辑器侧共同复用；不要为了旧 `Godot` 逻辑兼容而反向约束共享层设计。
- 若一段规则既可以做成纯 `Rust` 规则库，也可以做成 `Bevy system`，优先将“规则本身”下沉到共享 `Rust` 层，再由 `Bevy` 负责 `ECS` 调度与运行时集成。
- `Bevy` app 之间共享的运行时装配、定义到 `ECS` 的映射、定义到 `SimulationRuntime` seed 的集成，应优先放入共享 `Bevy` 层，而不是重复散落在各 app 本地模块中。

## 改动与迁移决策

功能与迁移决策默认都以 `Rust / Bevy` 为唯一长期实现方向：

- 默认规则：除非用户在需求中明确点名 `Godot`，否则 agent 默认只考虑 `Rust / Bevy` 路线；分析、方案、实现建议、代码定位与补丁目标都应优先落在 `bevy_server`、`game_bevy`、`game_core`、`game_data`、`game_protocol` 等 `Rust / Bevy` 侧，不以 `Godot` 代码作为默认分析入口、兼容基准或修补目标。
- 默认规则：功能开发默认只考虑并实现 `Rust / Bevy` 端；除非用户明确要求，否则不要在 `Godot` 端新增、补齐或对齐同类功能实现。
- 默认规则：凡是从 `Godot` 迁移功能到 `Rust` / `Bevy` 的任务，一律以 `Rust` 端实现为准，不需要为了旧 `Godot` 实现保留兼容层、双写逻辑、镜像逻辑或过渡适配。
- 若现有 `Godot` 代码与目标功能重复或冲突，应优先把能力迁到 `rust/`，而不是继续修补 `Godot` 版本。
- 若新增系统会长期存在，应优先放入 `rust/` 的共享层、协议层、核心层或 `Bevy` 运行时层，而不是继续堆积在 `Godot` 表现层。
- 若涉及数据格式调整，应优先直接落到共享 `Rust` 数据模型与协议定义，不以旧 `Godot` 数据结构为兼容基准。
- `Godot` 侧默认只承担渲染、UI、输入桥接与表现消费；当 `Rust` 端已有对应能力时，不应再在 `Godot` 侧重复实现一份。

当 agent 需要判断“某个系统是否应优先迁移到 `Rust / Bevy`”时，默认参考以下标准：

- 可脱离渲染与场景节点独立运行的逻辑，优先迁移。
- 需要稳定复用、可测试、可校验、可确定性执行的规则，优先迁移。
- 会被游戏运行时、服务端、编辑器或数据管线共同使用的结构与逻辑，优先迁移到共享 `Rust` 层。
- 强依赖 `Godot` 场景树、动画、输入、表现反馈的部分，可以暂时由 `Godot` 消费 `Rust` 结果，但不应把功能规则本身继续留在 `Godot` 侧。
- 如果一个功能仍处于快速试验期，但未来明确属于核心逻辑，也应优先在 `Rust` 侧建立最小实现，而不是先在 `Godot` 侧扩散。

当需求不明确时，agent 应按以下优先级做决策：

1. 优先实现或迁移到 `Rust` / `Bevy` 端，并以其为唯一功能基准。
2. 优先建设共享 `Rust` 基础层。
3. 优先减少逻辑、表现、编辑三者之间的新耦合。
4. 优先选择可验证、可回退的小步改动。

如果一个方案“短期方便但会强化 `Godot` 承载核心逻辑”或“要求为旧 `Godot` 功能保留兼容实现”，通常都不是优先方案。

## 编辑器与共享数据

围绕内容编辑能力的任务，默认按“以 `Rust` 共享层为权威、编辑器脱离 `Godot` 插件”思路处理：

- 不要再把新增编辑能力做在 `Godot` 插件内；如必须保留旧入口，也不应把它视为主实现。
- 共享的数据读取、写入、校验逻辑应尽量下沉到 `Rust` 侧复用。
- Web UI 相关能力应优先放入 `tools/tauri_editor`，而不是继续深绑 `Godot` 编辑器插件。

通信与数据应遵守以下原则：

- 不要为不同模块各自定义一套长期并存的数据结构。
- 不要引入与 `IPC / TCP` 主方案冲突的新长期通信主通道。
- 协议、消息、共享结构应尽量集中管理，避免散落在多个端内各自维护。

## 目录约定

仓库预期采用以下以 `Rust` 为中心的结构；agent 在创建新内容时应尽量遵守。

以下为核心目标结构示意，并非当前仓库的完整目录清单：

```text
cdc_survival_game/
├── project.godot
├── scenes/
├── core/
├── modules/
├── systems/
├── assets/
├── data/
├── addons/
├── tools/
│   └── tauri_editor/
├── rust/
│   ├── crates/
│   │   ├── game_data/
│   │   ├── game_bevy/
│   │   ├── game_protocol/
│   │   └── game_core/
│   └── apps/
│       ├── bevy_server/
│       └── bevy_debug_viewer/
```

目录使用建议：

- `rust/crates/game_data`: 共享数据模型
- `rust/crates/game_bevy`: `Bevy` app 共享运行时装配层
- `rust/crates/game_protocol`: IPC/TCP 协议定义
- `rust/crates/game_core`: 公共逻辑、校验、可复用规则
- `rust/apps/bevy_server`: `Bevy` 逻辑核心
- `rust/apps/bevy_debug_viewer`: `Bevy` 调试显示入口
- `tools/tauri_editor`: 独立内容编辑器
- `addons/`: 仅保留存量 `Godot` 插件，不再作为新增功能主落点

## 验证与交付

- 涉及 `rust/` 共享数据模型、协议或核心规则的改动，应优先补充最小可运行验证，例如单元测试、集成测试或 `cargo check` 级别验证。
- 涉及 `Godot` 表现层、UI、场景交互的改动，应至少说明采用了哪种运行验证、手动 smoke test 或最小复现方式。
- 默认规则：若用户没有明确要求编写测试，则不需要主动新增测试代码；优先使用现有测试体系、`cargo check`、运行验证或手动 smoke 说明完成交付。
- 若用户没有明确要求新增测试文件，不要擅自创建任何测试文件；应优先采用现有测试体系、最小验证命令或手动验证说明来完成交付。
- 除非用户明确指定需要编写测试脚本，否则开发功能时不必主动新增测试脚本；但仍应尽量提供可执行的最小验证方式或说明建议的验证步骤。
- 若本次任务因环境限制无法完成验证，agent 应明确说明未验证项、阻塞原因与建议的下一步验证方式。

agent 在提交改动、方案或说明时，尽量明确以下内容：

- 该改动属于哪个阶段
- 它服务于三端分离目标的哪一部分
- 是否偏离了 `Rust` 端唯一实现基准
- 是否引入了新的跨端耦合
- 后续最自然的下一步是什么

## 禁止事项

- 不要把“未来属于 `Bevy` 的核心规则”继续深度耦合进 `Godot` 表现层。
- 不要让 `Bevy` 承担渲染或 UI 表现职责。
- 不要把共享 `Rust` 层直接写成只能服务 `Bevy ECS` 的专用实现，除非该部分本来就属于 `Bevy` 运行时。
- 不要让编辑器直接依赖运行时表现层实现核心数据编辑能力。
- 不要重复定义共享数据结构，只图短期开发方便。
- 不要为了兼容旧 `Godot` 功能实现而保留多套长期并存逻辑。
- 不要进行把功能再次写回 `Godot` 的回迁式实现。

## 一句话原则

所有工作都应服务于同一个方向：

把 `逻辑`、`表现`、`编辑` 三件事拆开，并以 `Rust / Bevy` 作为功能唯一实现基准。
